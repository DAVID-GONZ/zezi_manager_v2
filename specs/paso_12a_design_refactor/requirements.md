# Requisitos — paso_12a_design_refactor

## Contexto

El design system "Andes Minimal v2" tiene tres problemas estructurales que estorban
la evolución del producto:

1. **`styles.css` con 3392 líneas en un solo archivo.** Cualquier cambio obliga a
   leer 100% del archivo; los conflictos en git son recurrentes; encontrar dónde
   vive una regla es búsqueda manual.
2. **`tokens.py` y `styles.css` derivan a mano.** `tokens.py:8-11` lo admite por
   escrito; ya hay drift activo (`Layout.SIDEBAR_WIDTH = 220` vs `--sidebar-width: 180px`).
3. **Bugs CSS observables y código legacy conviviendo.**
   - `andes-sidebar.collapsed` aplica `transform: translateX(-100%)` (esconde el
     sidebar entero) cuando el contrato dice "colapsado = solo iconos" (`layout.py:7`).
   - `.andes-sidebar-item` usa `align-items: left` (`styles.css:198`) — valor inválido.
   - `_topbar_legacy` (`layout.py:290`) coexiste con `_topbar` nuevo.

## Alcance

- `src/interface/design/styles.css` — partir en módulos.
- `src/interface/design/styles/` — nueva carpeta con sub-archivos CSS.
- `src/interface/design/tokens.py` — sincronizar con CSS, corregir drift.
- `src/interface/design/theme.py` — actualizar la carga de CSS (concatenar módulos).
- `src/interface/design/layout.py` — eliminar `_topbar_legacy`, fix sidebar collapsed.
- `scripts/sync_tokens.py` — generador/verificador de tokens (NUEVO).
- `tests/unit/design/test_tokens_sync.py` — test que falla si hay drift (NUEVO).

**Fuera de scope:**
- Cambiar colores (eso es paso 12c).
- Cambiar estructura del menú (12e).
- Cambiar el patrón del sidebar a rail (12d).

## Requisitos funcionales

### R1 — Split de `styles.css` en módulos

`styles.css` deja de ser monolítico. Se transforma en `styles/index.css` que importa
módulos por responsabilidad:

```
src/interface/design/styles/
├── index.css           # @import de todos
├── tokens.css          # :root { ... } (única fuente de variables)
├── reset.css           # reset base, body, .nicegui-content
├── typography.css      # h1..h4, .text-*, fuentes
├── layout/
│   ├── sidebar.css
│   ├── topbar.css
│   └── content.css
├── components/
│   ├── buttons.css
│   ├── tables.css       # incluye ag-grid overrides
│   ├── dialogs.css
│   ├── badges.css
│   ├── forms.css
│   ├── cards.css
│   └── chips.css        # context_chip
└── domain/
    ├── asistencia.css
    └── desempeno.css
```

`ThemeManager.aplicar()` carga `styles/index.css` Y, dado que NiceGUI inyecta CSS
inline en `<style>`, **concatena** el contenido de los módulos (no usa `@import`
del navegador). Esto evita peticiones HTTP extra y mantiene compatibilidad con la
estrategia actual.

### R2 — `tokens.py` deja de duplicar valores

Dos opciones aceptadas:

**Opción A (recomendada): generador**
- `scripts/sync_tokens.py` lee `styles/tokens.css`, parsea las variables `--*`, y
  regenera `src/interface/design/tokens.py`.
- El archivo `tokens.py` se marca con cabecera `# AUTOGENERATED — do not edit manually`.
- El test `test_tokens_sync.py` ejecuta el generador en memoria y verifica que su
  salida coincide byte a byte con el archivo en disco. Si no coincide, falla con un
  mensaje claro: "Ejecuta `python scripts/sync_tokens.py` y vuelve a correr el test."

**Opción B (alternativa): test de espejo**
- `tokens.py` se sigue escribiendo a mano, pero `test_tokens_sync.py` parsea
  `tokens.css` y compara cada `Colors.X` / `Spacing.X` / `Layout.X` contra el valor
  CSS. Falla si hay drift.

El implementer elige A o B según viabilidad técnica (regex sobre CSS es no trivial
para gradientes y rgba()). Justifica la elección en `progress/impl_12a.md`.

### R3 — Eliminar `_topbar_legacy`

`layout.py:290-335` define `_topbar_legacy`. Hay que:

1. Buscar todos los llamadores de `app_layout(...)` con firma legacy (positional args).
2. Migrarlos al nuevo API (`app_layout(ctx, contenido, page_titulo=…, …)`).
3. Borrar `_topbar_legacy` y la rama que decide cuál topbar usar.

### R4 — Fix bug `sidebar.collapsed`

Actualmente:
```css
.andes-sidebar.collapsed {
  transform: translateX(-100%);  /* esconde TODO el sidebar */
  width: 220px;
}
```

Debe ser: en estado colapsado el sidebar muestra solo iconos, ancho =
`var(--sidebar-collapsed)` (58px), no se traslada. Las labels se ocultan vía
`opacity: 0`.

(Nota: este fix queda obsoleto al final del 12d cuando el sidebar se vuelve rail
fijo. Pero entre 12a y 12d el comportamiento debe ser correcto para no romper la
demo si David ejecuta init.py entre pasos.)

### R5 — Fix `align-items: left`

`styles.css:198`: `align-items: left;` es inválido. Cambiar a `align-items: center`
(equivalente al render actual del navegador, que ignora el valor inválido).

### R6 — Resolver drift Layout.SIDEBAR_WIDTH

- `tokens.py:238`: `SIDEBAR_WIDTH = 220`
- `styles.css:103`: `--sidebar-width: 180px`
- `styles.css:162`: `.andes-sidebar { width: 220px }` (hardcoded)

Tras este paso: una sola fuente. `--sidebar-width: 220px` (valor que efectivamente
se renderiza hoy), `.andes-sidebar { width: var(--sidebar-width) }`, `tokens.py`
generado.

### R7 — Sin regresión visual

Después del split + fixes, la app debe verse idéntica a hoy en todas las páginas
(salvo el sidebar colapsado que pasa de "escondido" a "icon-only", visible y correcto
por R4).

### R8 — Criterio de cierre

- `python init.py` verde.
- 715 tests verdes + 1 nuevo (`test_tokens_sync.py`).
- `grep -r "from .styles import\|@import" src/interface/design/styles/` muestra
  estructura modular consistente.
- `grep "_topbar_legacy" src/` retorna vacío.
- Smoke-test manual: las páginas que David elija no muestran cambios visuales.
